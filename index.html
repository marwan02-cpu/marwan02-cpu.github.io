Research: LLMs at the edge

★ Awarded the highest thesis mark in my school and cohort

My undergraduate thesis profiles the power consumption of Vision-Language Models running on the edge, implemented on a Raspberry Pi 5 with Ollama. Language models are compute-hungry and draw significant power, so IoT developers building LLM apps on the edge need to know their workload’s draw to size the right battery. The existing literature had no framework to capture that comprehensively, so I built one from scratch, designed to generalize to any model, inference engine, and device.

Gaps I found in existing methods

  • No subsystem visibility. Existing tools couldn’t isolate CPU vs. RAM draw, so you couldn’t tell which component dominated.
  • Unverified accuracy. No way to confirm measurements were actually correct.
  • No custom sampling rate. USB meters are locked to vendor rates, pushing researchers to read values by eye and inject human error.
  • Not cost-effective. Accurate, high-rate meters run $600–1,400, still skip CPU/RAM, and demand electrical-engineering know-how.
  • No real-time dashboard. With many hardware and software parts interacting, anomalies needed to be caught visually and early.

Three measurement methods, combined

Software-defined meter

Free, built into the RPi 5

Queries per-component power (CPU, RAM, WiFi, system rails) via vcgencmd. I parse the raw GPU output with regex into CSV. Supports a custom sampling rate and real-time reads.

IoT-based meter

$20, Shelly Plug US Gen4

Captures total device power over a WebSocket connection from my MacBook, logged to CSV. Cost-effective and real-time. (TypeScript)

Plug-based meter

Verification

A redundant onboard-display meter used to confirm the IoT meter is accurate and not faulty.

Engineering challenges I solved

  • Built my own dataset. Public datasets were inconsistent, so I attached a Raspberry Pi Camera Module 3 and captured 5,000+ images of a driveway with full control over angle, quality, and frequency.
  • Custom labeler. Built a tool in Lovable with key-bound labels for rapid tagging, and disabled mutually exclusive options (e.g. daytime locks out nighttime) to cut labeling errors.
  • Synchronized the meters. Stood up an HTTP server on the RPi 5 that signals the IoT meter to start, aligning timestamps across both data sources instead of triggering them by hand.
  • Real-time observability. Wired Prometheus and Grafana Cloud to stream per-subsystem and total device power live during experiments.
  • Concurrent profiling. Used Python multiprocessing and subprocess to run the profiler and Gemma 3 inference simultaneously; supports any open-source Ollama model.

The framework, end to end

  1. Collect the data to be power-profiled
  2. Choose an inference engine on the device under test
  3. Sync the device profiler with the subsystem profiler
  4. Run the right profiling method per platform (nvidia-smi, Intel RAPL, or vcgencmd)
  5. Collect the measurements
  6. Structure the data when it arrives unstructured
